# Claude Code 可进化编程智能体：全新融合版设计方案（稳定进化 + 反盲从 + 可信度加权）

---

## 0. 设计北极星

### 0.1 最终目标

构建一个能在真实开发任务中持续变强的编程智能体系统：

- **稳定进化**：负反馈（痛苦）→ 根因 → 原则 → 护栏（rules/hooks/tests）→ 回归验证。
- **反盲从**：用户意见进入候选池，但不能绕过硬约束与审计；系统能在必要时强烈劝阻。
- **可信度加权**：对“用户”和“子智能体”都建立可验证的可信度画像，决策时加权。
- **低上下文负担**：不把完整哲学塞进 CLAUDE.md；采用“短内核 + 模块化外壳 + 事件门禁”。

### 0.2 关键约束

- **不变量内核**必须短且总是可用（几十到百行级别）。
- **适配与演化**通过“参数调节 + 模块启停”实现，而不是随意改内核。
- **可审计**：任何原则变化、门禁变化、画像变化，都要有证据链与可回滚记录。

## 1. 总体架构：三层轨道 + 一个调度器

### 1.1 三层轨道

1) **不变量内核（Kernel Rules）**：
- 五步循环顺序
- 根因方法（动词/形容词、5 Whys、人/设计/假设）
- 演绎审计插入点（Diagnosis→Plan）
- 认识论谦逊（不确定就停下获取信息/查文档，不编造）
- 痛苦信号（禁止“道歉+快修”掩盖问题）
- 反脆弱/否定法红线（防崩溃硬约束）
- Above/Below line（对话层级导航）
- 非羞辱表达（去情绪化但保留护栏）

2) **可变外壳（Skills + Subagents + Path Rules）**：
- 按场景加载（前端/后端/infra），不污染全局。
- 把重推理拆给子智能体；把结构化产出固化为 skills。

3) **强制门禁（Hooks + Scripts + Tests）**：
- 把“必须做”做成事件驱动门禁。
- 脚本优先；仅在需要判断时才用 prompt hook。

### 1.2 调度器（Orchestrator）

主会话承担一个“系统调度器”角色（不是重推理专家）：

- 决策门控：硬约束 → 演绎审计 → 可信度加权。
- 强制委托：关键阶段必须委派到指定子智能体。
- 画像治理：维护用户画像与子智能体画像，决定听谁、何时反对。

## 2. 进化闭环：Pain → Root Cause → Principle → Guardrail → Regression

### 2.1 痛苦信号（Pain Signals）来源

痛苦不只来自“代码报错”，而是**任何强负反馈**：

- 编译/运行/测试失败
- lint/formatter/security scan 失败
- 工具调用失败（命令返回非 0）
- Reviewer 标记 Critical
- 用户指出错误/不满意/需求对齐失败
- 性能指标/SLO 不达标
- 审计不通过（公理/系统/否定任意失败）

### 2.2 三道门：记录门 / 晋升门 / 回归门

**A. 记录门（必须落盘）**
- 触发痛苦即写 `ISSUE_LOG.md`，包含：证据、复现、直接原因、根因、5Whys、分类（人/设计/假设）、原则候选、护栏建议。

**B. 晋升门（经验→原则）**
- 不是每条都进 PRINCIPLES。晋升条件（满足任一）：
  - 同类问题复现≥2
  - 高风险目录/安全红线
  - 可门禁化（hook/test/rule 能落实）
  - 属于 Via Negativa 硬约束

**C. 回归门（防退化）**
- 原则/门禁升级后必须跑最小回归（smoke 或守护用例）。

## 4. PROFILE.json：它怎么“用起来”（不是讲理念，是开关与门禁）

### 4.1 定位

`PROFILE.json` 是系统的“运行时配置开关”，主要由：

- **Hooks 脚本读取** → 决定是否放行写入、是否强制 Plan/Audit、要跑哪些测试。
- **关键 skills/agents 在阶段入口读取一次** → 决定输出深度与检查清单强度。

> 它不承担哲学解释，只承担“可执行阈值”。

### 4.2 建议字段（最小可用）

```json
{
  "audit_level": "medium",
  "risk_paths": ["src/server/", "infra/", "db/"],
  "gate": {
    "require_plan_for_risk_paths": true,
    "require_audit_before_write": true,
    "require_reviewer_after_write": true
  },
  "tests": {
    "on_change": "smoke",
    "on_risk_change": "unit",
    "on_release": "full"
  },
  "permissions": {
    "deny_skip_tests": true,
    "deny_unsafe_db_ops": true
  },
  "output": {
    "verbosity": "balanced",
    "show_evidence": true
  }
}
```

### 4.3 典型工作流（门禁示例）

- Claude 试图对 `src/server/auth.ts` 写入：
  - PreToolUse(Write/Edit) → `pre_write_gate.sh`
  - 脚本读 PROFILE：发现路径在 risk_paths + require_plan/audit=true
  - 检查 `docs/PLAN.md` 是否存在、`docs/AUDIT.md` 是否 PASS
  - 不满足 → 直接阻断写入，迫使回到 Planner/Auditor

## 5. PRINCIPLES.md 与 rules：治理与执行分离（最稳）

### 5.1 定位

- `PRINCIPLES.md`：**原则源代码**（完整、可读、可讨论、可审计、可标注适用范围/例外/来源）。
- `.claude/rules/`：**运行时规则**（短、硬、可执行；可按 paths 条件加载）。

### 5.2 为什么不直接用 rules 取代 PRINCIPLES？

- rules 会自动注入上下文，过长会稀释与冲突。
- PRINCIPLES 需要更丰富的治理信息（来源、例外、冲突、过期策略）。

### 5.3 推荐做法：原则“编译”到 rules

- 晋升门通过后：将可执行部分“提纯”写入 `.claude/rules/10-guardrails.md` 或 `rules/paths/*.md`
- PRINCIPLES 保留完整版本，rules 只保留运行时硬约束。

## 6. 子智能体体系：岗位化 + 强制委托 + 绩效画像

### 6.1 子智能体就是 Subagents 吗？

是。本方案的“多个智能体”均是 Claude Code 子智能体（subagents）。

### 6.2 如果主智能体不委托怎么办？（关键：不靠自觉）

**两种强制策略：**

1) **入口 Skill 把委托写死**：
- `/evolve-task` 作为入口，明确要求：Explorer→Diagnostician→Auditor→Planner→Implementer→Reviewer。

2) **Hooks 门禁阻断关键动作**：
- 没有 Plan/Audit 不能写高风险路径。
- 写入后自动触发 Reviewer。

> 结果：即使主会话“忘了委托”，它也会被门禁推回正确轨道。

### 6.3 六岗位子智能体（推荐最小集合）

1) Explorer：证据与定位
2) Diagnostician：根因（动词/形容词、5Whys、人/设计/假设）
3) Auditor：演绎审计（公理/系统/否定）
4) Planner：电影剧本计划 + metrics + rollback
5) Implementer：按计划改代码 + 跑命令
6) Reviewer：git diff 审查 + Critical/Warn/Suggest

### 6.4 子智能体可信度加权（Believability for Agents）

引入 `docs/AGENT_SCORECARD.json`：
- 每次任务结束记录：产出是否被采纳、是否通过测试、是否引入返工/严重问题、失败类型。
- 形成一个“分数/评级”，供调度器选择：
  - 高分 agent 优先
  - 近期失误增加则降权或改用更大模型/更严格提示

> 这把“可信度加权”从人扩展到了 agent：agent 也必须挣信用。

## 7. 用户画像与反盲从：听谁、何时劝阻

### 7.1 USER_PROFILE.json（用户可信度画像）

核心字段建议：
- `domains`: { frontend/backend/infra/security/... } 每域一个可信度分数
- `signals`: 历史建议的正确率、需求描述一致性、是否经常要求跳过测试/审计
- `interaction_mode`: 输出偏好（简洁/详细）、是否接受强烈劝阻

### 7.2 校准机制（不靠自报身份）

可信度来自可验证记录：
- 用户提出原因/方案 → Explorer 抓证据验证
- 用户给 patch → 最小实验（smoke/unit）验证
- 多次被验证支持 → 提升该域分数
- 多次被验证推翻 → 降权并记录原因（知识过时/忽略边界/误解需求）

### 7.3 反盲从的三层决策门（强制）

1) **硬约束（Via Negativa / 安全红线）**：直接否决。
2) **演绎审计（公理/系统/否定）**：不通过则不执行。
3) **可信度加权**：在 1、2 不冲突前提下，按用户与agent的可信度决定采用谁的方案。

### 7.4 强烈劝阻机制（当用户指挥高风险动作时）

当用户要求：跳过测试、改生产配置、不可逆 DB 操作、绕过审计等：

- 必须给出反对理由（基于硬约束与风险）
- 必须给替代方案（更安全路径：feature flag、shadow write、渐进发布、回滚）
- 必须要求验证（最小回归）

## 9. Hooks：为什么选 Stop？Stop vs PreCompact 的职责分工

### 9.1 Stop hook（每轮稳定落盘点）

- 适合做：
  - 若触发 pain flag，则写 ISSUE_LOG
  - 更新 USER_PROFILE / AGENT_SCORECARD
  - 提议晋升原则并更新 PROFILE

Stop 的优点：**频繁且稳定**（每次回答结束都可能触发），不会因“没发生 compact”而漏记。

### 9.2 PreCompact（压缩前检查点）

- 适合做：
  - 在上下文即将被压缩前，做一次 checkpoint：把关键状态摘要写入文件，防止压缩导致遗忘。

### 9.3 组合策略（推荐）

- Stop：事件级记录（痛苦才写，减少噪音）
- PreCompact：压缩前统一 checkpoint（无论是否痛苦）
- PostToolUse：写入后跑自动检查与审查
