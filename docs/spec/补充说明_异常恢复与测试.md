# 可进化编程智能体：补充说明

本文档是对原设计方案的补充修正，解决以下两个核心问题：
1. **异常与中断恢复机制**
2. **测试策略与透明度**

---

## 1. 异常与中断恢复机制

### 1.1 问题背景

原设计假设流程总是完整执行，但现实中存在多种中断场景：
- 用户 Ctrl+C 手动终止（`Stop` hook 不会触发）
- 上下文压缩导致遗忘
- Subagent 失败但主会话继续
- 网络/系统异常导致会话中断

### 1.2 断点恢复机制

**新增 `SessionStart` hook（session_init.sh）**，在每次会话启动时：

1. **检测未处理的 pain flag**
   - 如果 `docs/.pain_flag` 存在，说明上次任务异常中断
   - 提示用户运行 `/evolve-task --recover` 完成诊断

2. **检测进行中的计划**
   - 如果 `docs/PLAN.md` 的 STATUS 为 `IN_PROGRESS`
   - 提示用户继续执行或重置计划

3. **恢复上下文**
   - 显示当前 PROFILE 配置
   - 显示最近一条 Issue

### 1.3 Subagent 分支逻辑

线性流程 `Explorer → Diagnostician → Auditor → Planner → Implementer → Reviewer` 必须处理失败分支：

| 失败点 | 处理方式 | 最大重试 |
|--------|----------|----------|
| Auditor 返回 FAIL | 回到 Diagnostician 补充根因分析 | 2 次 |
| Reviewer 发现 Critical | 回到 Planner 修订计划 | 2 次 |
| Implementer 执行失败 | 写 pain flag，下次 SessionStart 提示恢复 | - |

**在 `/evolve-task` Skill 中添加分支说明：**

```markdown
## Step 4: 委派 Auditor（演绎审计）
...

### 分支处理（必须遵守）
- 若 RESULT = FAIL：
  1. 将 Must-fix 列表写入 docs/AUDIT.md
  2. 回到 Step 3 重新委派 Diagnostician，要求补充根因
  3. 最多重试 2 次，若仍 FAIL 则写入 docs/DECISIONS.md 并请求用户介入
- 若 RESULT = PASS：继续 Step 5
```

### 1.4 上下文压缩保护

**PreCompact hook 增强（precompact_checkpoint.sh）**：

写入更完整的检查点信息：
- 当前任务目标
- 关键决策摘要
- 进行中的步骤（如果有）

**关键提醒**：压缩后 Claude **不会**自动读取 CHECKPOINT.md，需要在 `/evolve-task` 入口添加：
```markdown
## Step 0: 恢复上下文（强制）
- 读取 docs/CHECKPOINT.md 的最后一条
- 读取 docs/ISSUE_LOG.md 的最近 3 条
- 读取 docs/DECISIONS.md 的最近决策
```

## 2. 测试策略与透明度

### 2.1 组件可测试性分析

| 组件 | 可测试性 | 测试方法 |
|------|----------|----------|
| Hooks 脚本 | ✅ 高 | 独立运行，模拟 JSON 输入 |
| Rules | ❌ 低 | 只能观察行为变化 |
| Skills | ❌ 低 | 依赖 Claude 解释执行 |
| Subagents | ⚠️ 中 | Headless 模式 E2E |

### 2.2 Hooks 单元测试（TDD 可行）

创建 `test_hooks.sh`：

```bash
#!/bin/bash
set -e
cd "$(dirname "$0")/.."

echo "=== Hooks 单元测试 ==="

# 准备测试环境
mkdir -p docs
echo '{"audit_level":"medium","risk_paths":["src/server/"],"gate":{"require_plan_for_risk_paths":true,"require_audit_before_write":true}}' > docs/PROFILE.json

# Test 1: 门禁阻断（无 PLAN）
echo "Test 1: pre_write_gate blocks without PLAN"
rm -f docs/PLAN.md docs/AUDIT.md
INPUT='{"tool_name":"Write","tool_input":{"file_path":"src/server/test.ts"}}'
if echo "$INPUT" | ./.claude/hooks/pre_write_gate.sh 2>&1; then
  echo "FAIL: Should have blocked (exit 2)"
  exit 1
fi
echo "PASS"

# Test 2: 门禁阻断（AUDIT 非 PASS）
echo "Test 2: pre_write_gate blocks without AUDIT PASS"
echo "STATUS: READY" > docs/PLAN.md
echo "RESULT: FAIL" > docs/AUDIT.md
if echo "$INPUT" | ./.claude/hooks/pre_write_gate.sh 2>&1; then
  echo "FAIL: Should have blocked"
  exit 1
fi
echo "PASS"

# Test 3: 门禁放行
echo "Test 3: pre_write_gate allows with PLAN + AUDIT PASS"
echo "RESULT: PASS" > docs/AUDIT.md
if ! echo "$INPUT" | ./.claude/hooks/pre_write_gate.sh 2>&1; then
  echo "FAIL: Should have allowed"
  exit 1
fi
echo "PASS"

# Test 4: 非风险路径不阻断
echo "Test 4: pre_write_gate allows non-risk paths"
rm -f docs/PLAN.md docs/AUDIT.md
INPUT='{"tool_name":"Write","tool_input":{"file_path":"README.md"}}'
if ! echo "$INPUT" | ./.claude/hooks/pre_write_gate.sh 2>&1; then
  echo "FAIL: Should have allowed non-risk path"
  exit 1
fi
echo "PASS"

echo "=== 所有测试通过 ==="
```

### 2.3 E2E 验收（Headless 模式）

使用 Claude Code Headless 模式自动化验收：

```bash
#!/bin/bash
# e2e_test.sh

echo "=== E2E 验收测试 ==="

# 场景 1：门禁阻断
echo "Test: 门禁应阻断未授权写入"
rm -f docs/PLAN.md docs/AUDIT.md
RESULT=$(claude --headless -p "请修改 src/server/auth.ts 添加一行注释" 2>&1)
if echo "$RESULT" | grep -q "Blocked"; then
  echo "PASS: 门禁正确阻断"
else
  echo "FAIL: 门禁未生效"
  exit 1
fi

# 场景 2：门禁放行
echo "Test: 门禁应放行已授权写入"
echo "STATUS: READY" > docs/PLAN.md
echo "RESULT: PASS" > docs/AUDIT.md
RESULT=$(claude --headless -p "请修改 src/server/auth.ts 添加一行注释" 2>&1)
if echo "$RESULT" | grep -q "Blocked"; then
  echo "FAIL: 门禁错误阻断"
  exit 1
else
  echo "PASS: 门禁正确放行"
fi
```

### 2.4 审计日志（增强透明度）

添加全量工具调用日志：

```bash
# audit_log.sh - PostToolUse 全量记录
INPUT="$(cat)"
TOOL="$(echo "$INPUT" | jq -r '.tool_name')"
FILE="$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')"
TS="$(date -Iseconds)"

echo "[$TS] $TOOL: $FILE" >> docs/AUDIT_TRAIL.log
```

## 3. 配置修正清单

基于 Claude Code 官方文档，以下配置需要修正：

### 3.1 Hooks 变量引号

**错误**：
```json
"command": "${CLAUDE_PROJECT_DIR}/.claude/hooks/pre_write_gate.sh"
```

**正确**：
```json
"command": "\"$CLAUDE_PROJECT_DIR\"/.claude/hooks/pre_write_gate.sh"
```

### 3.2 Subagent permissionMode

**错误**（implementer.md）：
```yaml
permissionMode: plan  # plan 是只读模式！
```

**正确**：
```yaml
permissionMode: acceptEdits  # 允许文件编辑
```

### 3.3 新增必要 Hooks

| Hook 事件 | 脚本 | 用途 |
|-----------|------|------|
| SessionStart | session_init.sh | 断点恢复、上下文初始化 |
| SubagentStop | subagent_complete.sh | 子智能体绩效记录 |

## 5. 实施优先级

| 优先级 | 修正项 | 影响 |
|--------|--------|------|
| P0 | Hooks 引号修正 | 脚本无法执行 |
| P0 | implementer permissionMode | 无法写文件 |
| P1 | SessionStart 断点恢复 | 中断后无法恢复 |
| P1 | Subagent 分支逻辑 | 失败时死循环 |
| P2 | 测试套件 | 无法验证行为 |
| P2 | 审计日志 | 透明度不足 |
