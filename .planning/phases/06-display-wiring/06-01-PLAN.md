## Context

Phase 5 added `workflowFunnels: WorkflowFunnelOutput[]` to `RuntimeSummaryService.getSummary()` output when the `funnels` Map is provided. However, `buildEnglishOutput` and `buildChineseOutput` in `evolution-status.ts` do NOT render this data — they only show the hardcoded `heartbeatDiagnosis` block.

Phase 6 goal: actually render YAML-driven funnel stages in `/pd-evolution-status` output, in YAML-defined stage order.

## Implementation

### Task 1: Render workflowFunnels in buildEnglishOutput/buildChineseOutput

**Files:** `packages/openclaw-plugin/src/commands/evolution-status.ts`

**Read first:**
- `packages/openclaw-plugin/src/commands/evolution-status.ts` (lines 52-112 for `buildEnglishOutput`, lines 115-172 for `buildChineseOutput`)
- `packages/openclaw-plugin/src/service/runtime-summary-service.ts` (types: `WorkflowFunnelOutput`, `WorkflowFunnelStageOutput`)
- `.planning/phases/02-workflow-watchdog/workflows.yaml` (actual funnel/stage definitions for label reference)

**Action:**

In `buildEnglishOutput`, after the existing `// D: Heartbeat Diagnostician chain` block (around line 88), add:

```typescript
// E: YAML-driven Workflow Funnels (Phase 6)
if (summary.workflowFunnels && summary.workflowFunnels.length > 0) {
  lines.push('');
  for (const funnel of summary.workflowFunnels) {
    lines.push(`Workflow Funnel: ${funnel.funnelLabel}`);
    for (const stage of funnel.stages) {
      lines.push(`  - ${stage.label}: ${stage.count}`);
    }
  }
}
```

In `buildChineseOutput`, after the corresponding `// D:` block (around line 148), add the Chinese equivalent:

```typescript
// E: YAML驱动的Workflow Funnels (Phase 6)
if (summary.workflowFunnels && summary.workflowFunnels.length > 0) {
  lines.push('');
  for (const funnel of summary.workflowFunnels) {
    lines.push(`Workflow 漏斗: ${funnel.funnelLabel}`);
    for (const stage of funnel.stages) {
      lines.push(`  - ${stage.label}: ${stage.count}`);
    }
  }
}
```

**Acceptance criteria:**
- `summary.workflowFunnels` with valid data → funnel block appears in output
- Empty `workflowFunnels` array → funnel block skipped (no crash)
- `workflowFunnels` undefined → funnel block skipped (backward compat)
- Stage order matches YAML definition order (no hardcoded sequence)

### Task 2: Add degraded status when YAML is missing/invalid

**Files:** `packages/openclaw-plugin/src/service/runtime-summary-service.ts`

**Read first:**
- `packages/openclaw-plugin/src/service/runtime-summary-service.ts` (the `metadata.status` assignment logic around line 307-309)

**Action:**

The `metadata.status` in `getSummary()` is currently set based on GFI warnings. After Phase 5, we already set `status = 'degraded'` when `statsField` paths are missing. However, we need to also handle the case where `loaderWarnings` indicates YAML parse errors or file missing.

In the warnings-building block, also push to warnings when `loaderWarnings` is non-empty, AND update the status computation:

```typescript
// DEGRADED-01: If YAML had load errors, status must be degraded
if (options?.loaderWarnings && options.loaderWarnings.length > 0) {
  for (const w of options.loaderWarnings) {
    pushWarning(warnings, `YAML load warning: ${w}`);
  }
}
```

This should be added near the `workflowFunnelsOutput` warning block (around line 278 in the modified file).

**Acceptance criteria:**
- When `workflows.yaml` is missing → `metadata.status = 'degraded'` AND `metadata.warnings` contains "YAML load warning"
- When `workflows.yaml` is malformed → same degraded behavior
