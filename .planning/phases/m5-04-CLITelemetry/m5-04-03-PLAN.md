### Task 3: Emit diagnostician_artifact_commit_failed on commit exception

<read_first>
- packages/principles-core/src/runtime-v2/runner/diagnostician-runner.ts (succeedTask method)
</read_first>

<action>
Wrap the `committer.commit()` call in a try-catch that emits `diagnostician_artifact_commit_failed` when commit throws, then re-throws.

Current succeedTask calls commit like this:
```typescript
const commitResult = await this.committer.commit({
  runId: ctx.runId,
  taskId: ctx.taskId,
  output: ctx.output,
  idempotencyKey: `${ctx.taskId}:${ctx.runId}`,
});
```

Replace with:
```typescript
let commitResult;
try {
  commitResult = await this.committer.commit({
    runId: ctx.runId,
    taskId: ctx.taskId,
    output: ctx.output,
    idempotencyKey: `${ctx.taskId}:${ctx.runId}`,
  });
} catch (commitErr) {
  // Emit: diagnostician_artifact_commit_failed (TELE-02)
  this.emitDiagnosticianEvent('diagnostician_artifact_commit_failed', ctx.taskId, {
    taskId: ctx.taskId,
    runId: ctx.runId,
    errorCategory: commitErr instanceof PDRuntimeError ? commitErr.category : 'artifact_commit_failed',
    errorMessage: commitErr instanceof Error ? commitErr.message : String(commitErr),
  });
  throw commitErr; // Re-throw so outer try/catch in run() handles it
}
```

This ensures the event is emitted before the error propagates to handleLeaseOrPhaseError. The event fires before re-throw, so we get telemetry even on failure paths.
</action>

### Task 4: Emit principle_candidate_registered per candidate

<read_first>
- packages/principles-core/src/runtime-v2/runner/diagnostician-runner.ts (succeedTask after commit success)
</read_first>

<action>
After `commitResult` is obtained and before the `diagnostician_artifact_committed` event, emit `principle_candidate_registered` per candidate:

```typescript
// Emit: principle_candidate_registered per candidate (TELE-03)
const principleRecs = ctx.output.recommendations.filter(r => r.kind === 'principle');
for (const rec of principleRecs) {
  // Candidate IDs are generated inside committer.commit(). We can infer count
  // from commitResult.candidateCount and emit with matching index.
  // Per D-04: event has { candidateId, kind, description, sourceRunId }
  this.emitDiagnosticianEvent('principle_candidate_registered', ctx.taskId, {
    candidateId: `commit-${commitResult.commitId}-candidate`, // placeholder approach
    kind: 'principle',
    description: rec.description,
    sourceRunId: ctx.runId,
  });
}
```

Wait — candidate IDs are generated inside `SqliteDiagnosticianCommitter.commit()` using `randomUUID()`. The runner does not have access to them after commit. The TELE-03 payload requires `candidateId`.

**Alternative approach:** Since candidateId is opaque to the runner, and the committer already persisted candidates, we emit with available information:
- The event confirms a principle candidate was registered
- candidateId can use a deterministic derivation: `${commitResult.commitId}:${index}`
- This is observable and unique within the commit

Or better: The `diagnostician_artifact_committed` event already carries `candidateCount` which tells the observer how many candidates were registered. The `principle_candidate_registered` event provides the per-candidate detail.

Emit one `principle_candidate_registered` event per principle recommendation with available fields:
```typescript
const principleRecs = ctx.output.recommendations.filter(r => r.kind === 'principle');
for (let i = 0; i < principleRecs.length; i++) {
  const rec = principleRecs[i];
  this.emitDiagnosticianEvent('principle_candidate_registered', ctx.taskId, {
    candidateIndex: i,
    commitId: commitResult.commitId,
    kind: 'principle',
    description: rec.description,
    sourceRunId: ctx.runId,
  });
}
```

Note: The exact payload format can be refined — the key requirement is that one event fires per candidate. The runner uses available data since candidate IDs are generated inside the committer.
</action>
