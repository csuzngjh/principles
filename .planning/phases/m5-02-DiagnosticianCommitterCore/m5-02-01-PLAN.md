# Plan m5-02-01: DiagnosticianCommitter Core

## Objective

Implement `DiagnosticianCommitter` ‚Äî the atomic commit interface that persists a `DiagnosticianOutputV1` to the artifact registry (artifacts + commits + principle_candidates) inside a single SQLite transaction, with idempotent re-commit support.

## Context

- **m5-01** (commit f88e9de2) created the three tables: `artifacts`, `commits`, `principle_candidates` with FKs and UNIQUE constraints
- **COMT-01** through **COMT-06** define the DiagnosticianCommitter interface and behavior
- Full design rationale in `m5-02-CONTEXT.md`

## Requirements

| ID | Description |
|----|-------------|
| COMT-01 | `DiagnosticianCommitter` interface with `commit(input: CommitInput): Promise<CommitResult>` |
| COMT-02 | Transaction-wrapped commit: artifact + commit + all candidates in ONE SQLite transaction |
| COMT-03 | Extract `kind='principle'` recommendations as `PrincipleCandidateRecord` rows |
| COMT-04 | Idempotent re-commit: UNIQUE on `commits.run_id` and `commits.idempotency_key` means re-commit returns existing result |
| COMT-05 | Commit failure returns `PDRuntimeError` with `errorCategory: 'artifact_commit_failed'` ‚Äî no partial rows |
| COMT-06 | `CommitResult` returns `{ commitId, artifactId, candidateCount }` on success |

## Tasks

### Task 1: Implement DiagnosticianCommitter interface + SQLite implementation

**Type:** auto

**Implementation file:** `packages/principles-core/src/runtime-v2/store/diagnostician-committer.ts`

**Interface:**
```typescript
export interface DiagnosticianCommitter {
  commit(input: CommitInput): Promise<CommitResult>;
}

export interface CommitInput {
  runId: string;
  taskId: string;
  output: DiagnosticianOutputV1;
  idempotencyKey: string;
}

export interface CommitResult {
  commitId: string;
  artifactId: string;
  candidateCount: number;
}
```

**Implementation (SqliteDiagnosticianCommitter):**
- Constructor takes `SqliteConnection`
- `commit()` method:
  1. Validate `input.output` via `Value.Check(DiagnosticianOutputV1Schema, input.output)` ‚Äî throw `input_invalid` if fails
  2. Begin IMMEDIATE transaction
  3. Generate `commit_id = crypto.randomUUID()`, `artifact_id = crypto.randomUUID()`
  4. Insert `artifacts` row with `artifact_kind = 'diagnostician_output'`, `content_json = JSON.stringify(input.output)`
  5. Insert `commits` row
  6. For each `recommendation` in `input.output.recommendations` where `kind === 'principle'`:
     - Generate `candidate_id = crypto.randomUUID()`
     - Derive `idempotency_key = \`${commit_id}:${index}\``
     - Extract `title` (from `recommendation.title` or `recommendation.principleId` or fallback to description truncated to 100 chars), `description`, `confidence`
     - Insert `principle_candidates` row
  7. COMMIT
  8. Return `{ commitId, artifactId, candidateCount }`

**Error handling:**
- On UNIQUE constraint failure: query existing rows, return found result (idempotent)
- On other SQL error: ROLLBACK, throw `PDRuntimeError{category: 'artifact_commit_failed', details: {originalError, constraint}}`
- On validation error: throw `PDRuntimeError{category: 'input_invalid', ...}`

**Verification:**
- All SQL statements use parameterized queries (`?` placeholders)
- No string interpolation in SQL
- Transaction uses `db.transaction()` wrapper from better-sqlite3
- Foreign key constraints enabled (pragma) ‚Äî any violation rolls back
