```typescript
/**
 * TrajectoryLocator — abstract interface for trajectory locate operations.
 *
 * A "trajectory" is a group of runs for a single task. The TrajectoryCandidate.trajectoryRef
 * maps to the taskId. When locating by runId, the locator finds the run's task_id,
 * then returns all runs for that task as a single trajectory.
 *
 * All trajectory locate operations go through this interface, enabling
 * swap between SQLite (default) and test doubles.
 */
import type { TrajectoryLocateQuery, TrajectoryLocateResult } from '../context-payload.js';

export interface TrajectoryLocator {
  /**
   * Locate trajectories matching the given query.
   *
   * The query may contain one or more criteria. Supported modes:
   *   - painId: exact match on run.run_id (treated as trajectory identifier)
   *   - taskId: find all runs for task via idx_runs_task_id
   *   - runId: find run, then return all runs for its task_id
   *   - timeRange: query runs.started_at BETWEEN start AND end, group by task_id
   *   - sessionId + workspace: PD-managed hints, workspace-scoped
   *   - executionStatus: filter runs by status via idx_runs_status (stretch)
   *
   * Returns TrajectoryLocateResult with candidates array.
   * No match → empty candidates, never throws.
   */
  locate(query: TrajectoryLocateQuery): Promise<TrajectoryLocateResult>;
}
```

**Implement `locate()` as a router** that checks query fields in priority order and delegates to private handler methods. Priority order:
1. `query.painId` → `locateByPainId(query.painId, query)` — exact match on runs.run_id
2. `query.taskId` → `locateByTaskId(query.taskId, query)` — find all runs via idx_runs_task_id
3. `query.runId` → `locateByRunId(query.runId, query)` — find run, get its task_id, return full trajectory
4. `query.timeRange` → `locateByTimeRange(query.timeRange, query)` — query runs.started_at BETWEEN, group by task_id
5. `query.sessionId && query.workspace` → `locateBySessionHint(query)` — workspace-scoped via join to tasks
6. No criteria match → return empty result with `candidates: []`

**Each handler returns `TrajectoryLocateResult`** with:
- `query`: the original query echoed back
- `candidates`: array of `TrajectoryCandidate` objects

**TrajectoryCandidate construction pattern:**

**SQL queries per mode:**

Mode 1 (painId — exact match on run_id):
```sql
SELECT task_id FROM runs WHERE run_id = ?
```


Mode 2 (taskId — direct lookup):
```sql

```

Mode 3 (runId — indirect lookup):
```sql
SELECT task_id FROM runs WHERE run_id = ?
```
Then fetch all runs for that task_id (same as Mode 1 step 2).

Mode 4 (timeRange — date range):
```sql
SELECT DISTINCT task_id FROM runs WHERE started_at >= ? AND started_at <= ?
```
For each task_id found, construct a TrajectoryCandidate with confidence=0.7, reasons=['date_range_match'].

Mode 5 (sessionId + workspace — session hint):
This requires a JOIN to tasks table because workspace isolation is task-scoped.
```sql
SELECT DISTINCT r.task_id FROM runs r
  JOIN tasks t ON r.task_id = t.task_id
  WHERE t.workspace_dir = ? OR t.input_ref LIKE ?
```
Actually, looking at the tasks table schema, there is NO workspace_dir or workspace column in the tasks table. The workspace is implicit via the SqliteConnection path (each workspace has its own state.db). So sessionId hint mode should:
- Filter runs by time range (if provided alongside sessionId)
- Group by task_id
- Set confidence=0.5 with reasons=['session_hint']
If workspace is provided but the connection is already workspace-scoped, it is a validation check only.

**WAIT — re-examine the schema.** The tasks table does NOT have workspace_dir. Looking at DiagnosticianTaskRecord, it has workspaceDir but that is a runtime-level field, not a DB column. The workspace isolation in M2 is achieved by each SqliteConnection pointing to a different `.pd/state.db` file per workspace. Therefore:
- Mode 5 (sessionId + workspace): The workspace parameter serves as a validation hint (confirm the connection matches), but the actual filtering uses whatever is available. Since sessionId is not a column in runs or tasks tables either, this mode returns all task_ids from recent runs with confidence=0.5, noting the limitation.
