  <action>
Implement the three-point registration flow for successful code candidates:

1. create ledger `Implementation(type=code, lifecycleState='candidate')`
2. write storage assets (`manifest.json`, `entry.js`)
3. append code-candidate lineage record

NOC-02 traceability must be explicit in both lineage and manifest-facing metadata. Record:

- `principleId`
- `ruleId`
- `sourceSnapshotRef`
- `sourcePainIds`
- `sourceGateBlockIds`
- `sourceSessionId`
- `artificerArtifactId`

If the implementation asset write fails after the ledger record is created, the code must deterministically clean up or archive that candidate so later replay/promotion commands do not see a half-created implementation.
  </action>
  <verify>
    <automated>cd packages/openclaw-plugin && npx vitest run tests/service/nocturnal-service-code-candidate.test.ts tests/core/nocturnal-artifact-lineage.test.ts</automated>
  </verify>
  <done>
    - Successful candidates appear in ledger as `candidate` implementations
    - Storage assets are written under the implementation root with manifest lineage fields populated
    - Lineage records explicitly preserve both pain refs and gate-block refs where present
    - Storage/ledger partial failure behavior is deterministic and tested
  </done>
  <acceptance_criteria>
    - Successful candidate writes create `Implementation(type=code, lifecycleState='candidate')`
    - Asset storage writes `manifest.json` and `entry.js` under the implementation asset root
    - Manifest stores lineage-facing metadata subordinate to the ledger lifecycle state
    - Lineage registration records source principle/rule/snapshot plus explicit pain refs and gate-block refs
    - On storage failure after ledger create, implementation state is cleaned up or archived deterministically
  </acceptance_criteria>
</task>
