<d08_note>
## D-08 Clarification: Per-Event-Type updateStats Branches vs. Dynamic Funnel Aggregation

Gemini raised a concern that the 6 updateStats() branches hardcode event-to-stats-field mapping, which could conflict with D-08's spirit of "WORKFLOW_FUNNELS drives stats aggregation."

**Resolution:** The per-event-type branches are the correct approach for Phase 02, for these reasons:

1. **Existing system pattern:** `updateStats()` already uses per-event-type `else if` branches for all 8 existing event types (pain_signal, diagnosis_task, rule_enforced, etc.). Adding 6 more branches is consistent with this established pattern.

2. **D-08 scope:** D-08 says WORKFLOW_FUNNELS is "used for stats aggregation" — meaning the YAML table DEFINES which events feed which stats. The per-event-type branch is the implementation of that aggregation decision.

3. **Future dynamic aggregation (post-Phase 02):** A future enhancement could make `updateStats()` consult `WorkflowFunnelLoader.getAllFunnels()` to resolve `eventType -> statsField` dynamically, eliminating hardcoded branches. This is explicitly out of scope for Phase 02.

4. **Type safety:** Hardcoded branches are validated by TypeScript at compile time. Dynamic resolution would require runtime lookups and add complexity without benefit in the current phase.

Therefore: Task 4 adds 6 `else if` branches exactly as originally planned. The `EvolutionStats` fields are explicit and type-safe.
</d08_note>
