## Codex Review

## Summary  The plan is directionally sound and has a reasonable wave structure: define loader and types first, then integrate event-log and emitters. The biggest risk is that the plans may satisfy the event additions but not fully satisfy the “YAML workflow funnel framework” goal. In particular, `WORKFLOW_FUNNELS`/`workflows.yaml` is not clearly integrated into stats aggregation, `workflows.yaml` itself is not listed as a planned artifact/config contract, and hot-update/error behavior is underspecified. Event emission is mostly well-scoped, but needs stronger guarantees around canonical emission points, payload privacy, duplicate events, and tests.  ## Strengths  - The dependency ordering is mostly correct: loader and event type expansion happen before `event-log.ts`, and emitters depend on event-log methods. - 02-01 correctly avoids coupling `WorkflowFunnelLoader` to `event-log.ts`, which keeps the loader reusable and reduces circular dependency risk. - 02-02 isolates type expansion, which is a good Wave 0 task and aligns with D-05 and D-07. - 02-03 centralizes recording and stats updates inside `event-log.ts`, matching D-06 and preserving the event-log as the canonical write path. - 02-04 and 02-05 separate Nocturnal and RuleHost emission work, which reduces merge conflict risk and makes functional verification easier. - The planned six event types map cleanly to PD-FUNNEL-2.3 and PD-FUNNEL-2.4.  ## Concerns  - **HIGH: D-08 is not clearly implemented.** 02-03 says it adds six hardcoded `updateStats()` branches and six `EvolutionStats` fields, but D-08 says `WORKFLOW_FUNNELS` is used for stats aggregation. If stats remain event-specific hardcoded branches, the funnel abstraction is not actually driving aggregation.  - **HIGH: D-01 is under-specified.** If `workflows.yaml` is the SSOT, the plan needs to define where it lives, its schema, how it maps to `WORKFLOW_FUNNELS`, and how runtime code behaves when it is missing or invalid. 02-01 only lists `package.json` and `src/core/workflow-funnel-loader.ts`, so the YAML contract itself is missing from the plan.  - **HIGH: The relationship between “WORKFLOW_FUNNELS pure in-memory definition table” and “workflows.yaml is SSOT” is ambiguous.** PD-FUNNEL-2.1 suggests an in-memory table, while D-01 says YAML is the source of truth. The plan should clarify whether `WORKFLOW_FUNNELS` is loaded from YAML into memory, a fallback constant, or a typed runtime projection.  - **MEDIUM: Hot update behavior lacks edge-case handling.** 02-01 mentions `watch()`, but does not specify debounce, partial-write handling, atomic reload behavior, invalid YAML fallback, watcher cleanup, duplicate watcher prevention, or whether consumers are notified after reload.  - **MEDIUM: Error handling for malformed YAML is missing.** A developer-maintained `.state/workflows.yaml` will eventually contain syntax or schema mistakes. The loader should preserve the last valid config, log validation errors, and avoid crashing the process.  - **MEDIUM: Test coverage is absent from all plans.** The plans mention implementation files only. This phase has enough behavior to require tests for YAML load/reload, invalid config, event recording methods, stats aggregation, and emission points.  - **MEDIUM: Event payload privacy/security is not addressed.** `code_candidate_created` and artifact-related events can accidentally log code snippets, file contents, absolute paths, prompts, or agent-private context. EventData interfaces should explicitly constrain payloads to IDs, paths relative to workspace if safe, hashes, counts, and metadata needed for observability.  - **MEDIUM: RuleHost event volume and duplication risk are unclear.** `rulehost_evaluated` emitted “always” may generate high-volume logs. It also needs clear semantics when a rule both blocks and requires approval, or when multiple rules evaluate in one gate run.  - **MEDIUM: `requireApproval` casing is risky.** The requirement uses `requireApproval`, while the event naming pattern likely uses lowercase snake case. The plan should pin the exact event literal, for example `rulehost_require_approval` or `rulehost_requireApproval`, and ensure consistency across `EventType`, `recordXxx()`, stats, and queries.  - **MEDIUM: 02-03 edits `src/types/event-types.ts` after 02-02.** This is acceptable because it depends on 02-02, but the scope should be narrowed. If 02-03 needs to add `EvolutionStats` fields in the same file, say that explicitly; otherwise it risks reworking Wave 0 changes.  - **LOW: The loader API may be too narrow.** `getStages()` and `getAllFunnels()` are useful, but stats aggregation may also need event-to-funnel lookup, stage ordering, category mapping, and validation metadata.  - **LOW: No migration/backward compatibility behavior is specified.** If existing event logs/stats exist, adding six stats fields should preserve old data and default missing fields safely.  - **LOW: Operational observability of the loader itself is missing.** Since this phase is about workflow observability, the loader should expose last load time, active config version/hash, and last validation error, at least internally or via logs.  ## Suggestions  - Add an explicit `workflows.yaml` contract to 02-01: path, schema, required fields, example structure, allowed event names, stage ordering, and validation rules.  - Make `WORKFLOW_FUNNELS` a runtime projection loaded from `workflows.yaml`, not an independent competing source of truth. If a fallback constant exists, document that YAML still wins and the fallback is only for safe startup.  - Change 02-03 so stats aggregation is funnel-driven: event type resolves to funnel/stage from `WorkflowFunnelLoader`, then updates generic funnel counters instead of only six bespoke branches.  - Define loader failure semantics: missing YAML, malformed YAML, schema-invalid YAML, file deleted during reload, and watcher errors should all keep the last known-good config and surface a diagnostic.  - Add tests as first-class plan files. Minimum useful coverage: valid YAML load, invalid YAML preserves last valid config, watcher reload updates stages, `recordXxx()` persists expected event type/data, stats aggregation uses funnel definitions, Nocturnal emits exactly three events from canonical points, RuleHost emits evaluated/blocked/approval events without duplicates.  - Constrain all six `EventData` interfaces to observability-safe fields. Prefer IDs, timestamps, rule names, stage names, status, relative paths, hashes, counts, and reason codes. Avoid raw code, prompts, full artifacts, absolute user paths, or environment data.  - Clarify RuleHost semantics: one `rulehost_evaluated` per gate run or per rule, whether blocked and approval events can co-occur, and what correlation ID links related events.  - Add correlation fields across workflow events, such as `workflowRunId`, `agentId`, `workspaceId`, `taskId`, or equivalent existing identifiers. This is important for funnel analysis.  - Specify exact event literals and categories in 02-02 before implementation. This prevents drift between requirement names, type names, record method names, and stats keys.  - Ensure `dispose()` is verified. Watchers are easy to leak in long-running agent processes, especially with hot reload.  ## Risk Assessment  Overall risk: **MEDIUM**.  The phase is achievable and the wave split is sensible, but the current plans are more complete for “add six events” than for “YAML workflow funnel framework.” The main delivery risk is architectural: if stats are implemented as hardcoded event branches and the YAML loader remains isolated, the phase will not actually satisfy D-01 and D-08. The main runtime risks are malformed hot-reload config, noisy or duplicated RuleHost events, and accidentally logging sensitive code/artifact data. These are fixable with a tighter YAML contract, funnel-driven aggregation design, explicit event semantics, and tests.
