## Gaps Being Closed

### Gap 1: CR-01 -- recordFalsePositive should be recordTruePositive

**File:** `packages/openclaw-plugin/src/hooks/prompt.ts`
**Line:** 362

**Current state:** prompt.ts line 362 calls `learner.recordFalsePositive(correctionCue)`. The comment says "record it as a false positive so the weighted scoring formula has real data."

**Why wrong:** A correction cue detection means the keyword matcher correctly identified a dissatisfaction pattern. This is a confirmed match = true positive. Calling recordFalsePositive applies x0.8 weight decay (correction-cue-learner.ts line 234: `keyword.weight = keyword.weight * 0.8`) and increments FP counters. Over time, correctly-matching keywords decay toward zero, suppressing the very patterns the system is designed to detect.

**Required state:** Change `learner.recordFalsePositive(correctionCue)` to `learner.recordTruePositive(correctionCue)` on line 362. This increments `truePositiveCount` (line 210) instead of `falsePositiveCount` (line 230) and does NOT apply weight decay.

**Design context:** D-39-07 says "when user expresses dissatisfaction, call recordFalsePositive()" but the code review CR-01 identifies this as semantically backwards. The correction cue DETECTION is a true positive. The false positive case would be if the keyword matched but the user was NOT actually expressing dissatisfaction (which does not happen here since detectCorrectionCue only returns a match when the user IS expressing dissatisfaction). The D-39-07 decision conflated "user expressing dissatisfaction" (negative sentiment) with "false positive" (incorrect keyword match). These are orthogonal: a user can express dissatisfaction AND the keyword match can be correct (true positive).

### Gap 2: keyword_optimization trigger unreachable in default config

**File:** `packages/openclaw-plugin/src/service/evolution-worker.ts`
**Lines:** 2396-2412

**Current state:** The keyword_optimization trigger (lines 2397-2403) is inside `if (sleepConfig.trigger_mode === 'periodic')` block (line 2396). Default `trigger_mode` is `'idle'` (nocturnal-config.ts line 30), so the keyword_optimization code is never reached.

**Required state:** Extract the keyword_optimization periodic check OUTSIDE the trigger_mode conditional, so it fires in both 'idle' and 'periodic' modes. The keyword_optimization has its own dedicated config (`kwOptConfig.period_heartbeats=24`, `kwOptConfig.enabled`) and should operate independently.
