<objective>
Close the final boundary-trust gaps in the nocturnal production path.

This plan removes runtime capability guessing, time-bounds session selection, and adds end-to-end contract tests that prove the main path writes to the right workspace and preserves the right session context.
</objective>

<must_haves>
- Runtime availability no longer depends on `constructor.name === 'AsyncFunction'`
- Background workflow state distinguishes pre-launch runtime unavailability from downstream execution failure
- Sleep-reflection / nocturnal candidate selection cannot choose sessions newer than the triggering pain/task time
- End-to-end tests prove:
  - pain context preserves `session_id`
  - command/hook writes stay inside active workspace `.state`
  - bounded selection rejects or skips future-session drift
</must_haves>

<tasks>
1. Replace runtime availability guessing with an explicit contract.
   - Refactor `subagent-probe.ts`
   - Remove constructor-name semantics from capability detection and debug logic
   - Preserve a clean distinction between "callable runtime exists" and "runtime call failed"

2. Add bounded session selection.
   - Introduce a trigger-time upper bound in the fallback candidate search path
   - Ensure pain-session exact match still wins
   - Ensure no bounded candidate leads to explicit failure or skip, not silent reassignment

3. Align manual/runtime entry points.
   - Verify `/pd-reflect` and related trigger paths use the same runtime/workspace contract semantics as worker-driven nocturnal flow
   - Remove any remaining divergence that would make manual success differ from background behavior

4. Add end-to-end contract tests.
   Minimum cases:
   - valid pain signal writes in active workspace and queue preserves `session_id`
   - hook/command write path stays under workspace `.state`, not HOME/cwd fallback
   - sleep reflection cannot select a session whose timestamp is after the triggering task/pain time
   - runtime-unavailable path is reported differently from workflow execution failure
</tasks>
