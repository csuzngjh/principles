# Phase 1.5: Cross-Domain Validation - Context

**Gathered:** 2026-04-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Stress-test SDK universality by implementing a CodeReviewPainAdapter that captures pain from code review events (diff complexity, comment sentiment, process violations), and validate that PainSignal + PainSignalAdapter handles this non-coding, human-judged domain without interface changes. Must exercise both pain capture AND principle injection end-to-end.

**Why Code Review:** Unlike tool failures (clear signal), review feedback is human-judged, ambiguous, emotionally-charged, and high-cardinality. It is the most extreme non-coding case — it will reveal whether PainSignalSchema is truly universal or just "coding-shaped."

</domain>

<decisions>
## Implementation Decisions

### Extreme Domain
- **D-01:** Code Review Feedback 作为极端测试域。高基数人类判断数据（diff 复杂度、评论情感、过程违规），与 coding 域差异极大。无工具调用，痛点来自审查质量而非系统错误。

### CodeReviewPainAdapter — Pain Trigger Definition (Claude's Discretion)
- **D-02:** Pain triggers 三类：
  1. **Diff complexity signals** — 大面积 diff (>500 行)、多文件改动、循环依赖、深层嵌套。Pain score = file_count × avg_complexity
  2. **Comment sentiment signals** — 负面评论语气（frustration/dismissiveness）、同一文件重复纠正、超过 3 天未解决的 thread。Pain score = negative_sentiment × recency_days
  3. **Process violation signals** — PR 无测试、breaking change 无通知、安全漏洞引入。Pain score = severity × affected_scope

### Context Fields (Full Thread + Sentiment)
- **D-03:** PainSignal context 包含：authorId, reviewerIds, affectedFiles[], diffSize, fileTypes[], commentCount, unresolvedThreadCount, avgSentiment, reviewAge, painScore

### Pain Capture + Injection E2E Validation
- **D-04:** 必须测试完整 E2E 流程：CodeReviewEvent → PainSignalAdapter.capture() → PainSignal → DefaultPrincipleInjector.getRelevantPrinciples() → formatForInjection()。Pain capture alone is insufficient.

### Success Criteria (Claude's Discretion)
- **D-05:** 成功标准：PainSignalAdapter<ReviewEvent> 产生有效 PainSignal，无需修改 PainSignal 或 PainSignalAdapter 接口。PainSignalSchema 通过验证。
- **D-06:** SDK 三个域（coding + writing + code review）均产生有效 PainSignal = 验证完成

### API Change Policy
- **D-07:** API 冻结原则：任何 review 域暴露的 gap 必须通过 context 扩展解决，不改 PainSignal 核心 schema。如需新字段，以 optional 字段添加，不做 breaking change。

### Claude's Discretion
- CodeReviewEvent 的具体 TypeScript 接口定义
- Pain score 的具体算法（score = f(sentiment, complexity, age)）
- 情感分析策略（基于关键词 vs 外部服务）
- Test data 的构造策略（模拟 review 事件）

</decisions>

<specifics>
## Specific Ideas

- ReviewEvent interface: { prId, authorId, reviewerIds, filesChanged[], diffLines, comments[], createdAt, reviewAge, sentimentScore }
- Pain score algorithm: `score = Math.round(0.4 * processViolationScore + 0.35 * sentimentScore + 0.25 * complexityScore)`
- Sentiment scoring: 基于关键词（frustrat*, dismiss*, "this is wrong" → negative; "lgtm", "nice" → positive）
- Test data: 构造 5-8 个模拟 ReviewEvent，从简单到复杂覆盖三个 trigger 类型
- Gap-if: 如果 PainSignalSchema 验证失败（missing 某字段），则以 PainSignalSchema.create() 的 defaults 扩展，不改 schema

</specifics>
