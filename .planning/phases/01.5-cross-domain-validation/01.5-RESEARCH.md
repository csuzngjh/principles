## 1. ReviewEvent Interface Draft

```typescript
// packages/principles-core/src/adapters/code-review/review-event-types.ts

/**
 * Code review event types for principles-core.
 *
 * Per D-01: Code review is an extreme non-coding domain.
 * Pain signals come from human judgment, not tool failures.
 */

/**
 * Types of pain triggers in code review.
 */
export type ReviewPainTriggerType =
  | 'diff_complexity'      // Large/hard diffs
  | 'negative_sentiment'  // Hostile or dismissive comments
  | 'process_violation';  // Policy breaches (no tests, security issues)

/**
 * Represents a single review comment or comment thread.
 */
export interface ReviewComment {
  id: string;
  authorId: string;
  body: string;              // Raw comment text
  sentimentScore: number;    // -100 (negative) to +100 (positive)
  createdAt: string;         // ISO 8601
  resolvedAt?: string;        // ISO 8601, undefined if unresolved
  filePath?: string;         // File under review (if line comment)
}

/**
 * File changed in a PR/review.
 */
export interface ChangedFile {
  path: string;
  linesAdded: number;
  linesDeleted: number;
  // Simple cyclomatic-complexity proxy (higher = more complex)
  changeMagnitude: number;   // linesAdded + linesDeleted
}

/**
 * Structured code review event from a review platform webhook
 * or synthetic test data.
 */
export interface ReviewEvent {
  // Identification
  prId: string;
  repositoryId: string;

  // Participants
  authorId: string;           // PR author
  reviewerIds: string[];       // Assigned reviewers

  // Diff data
  filesChanged: ChangedFile[];
  totalLinesAdded: number;
  totalLinesDeleted: number;

  // Comment data
  comments: ReviewComment[];
  unresolvedThreadCount: number;

  // Temporal
  createdAt: string;           // ISO 8601 when review started
  reviewAgeDays: number;       // How long since first review comment

  // Optional: PR metadata
  hasTests: boolean;
  hasSecurityReview: boolean;
  isBreakingChange: boolean;
  labels: string[];

  // Required: session/trace linkage
  sessionId: string;
  traceId?: string;
}
```

## 2. Pain Score Algorithm (3-Signal Model)

Following D-02 from 01.5-CONTEXT.md, the pain score is a weighted composite of three independent signals:

```
score = Math.round(0.40 * processViolationScore
                  + 0.35 * sentimentScore
                  + 0.25 * complexityScore)
```

### Signal 1: Diff Complexity Score (weight: 0.25)

```typescript
function deriveComplexityScore(event: ReviewEvent): number {
  const fileCount = event.filesChanged.length;
  const totalChanges = event.totalLinesAdded + event.totalLinesDeleted;

  // file_count × avg_complexity proxy
  const avgMagnitude = fileCount > 0
    ? event.filesChanged.reduce((sum, f) => sum + f.changeMagnitude, 0) / fileCount
    : 0;

  const complexity = fileCount * avgMagnitude;

  // Normalize to 0-100
  // Rough thresholds:
  //   < 500      → low (small PR)
  //   500-2000  → medium
  //   2000-5000 → high
  //   > 5000    → critical
  if (complexity < 500)  return Math.round(complexity / 500 * 40);       // 0-40
  if (complexity < 2000) return Math.round(40 + (complexity - 500) / 1500 * 25);  // 40-65
  if (complexity < 5000) return Math.round(65 + (complexity - 2000) / 3000 * 20); // 65-85
  return Math.min(100, 85 + (complexity - 5000) / 5000 * 15);
}
```

### Signal 2: Comment Sentiment Score (weight: 0.35)

```typescript
const NEGATIVE_KEYWORDS = [
  'wrong', 'broken', 'bad', 'terrible', 'awful',
  'frustrat', 'annoy', 'dismiss', 'reject',
  'this is a problem', 'lgtm but', 'almost',
];

const FRUSTRATION_MARKERS = [
  'again', 'still', 'already', 'repeated',
  'why not', 'should have', 'never',
];

function deriveSentimentScore(event: ReviewEvent): number {
  const negativeComments = event.comments.filter(c => {
    const body = c.body.toLowerCase();
    return NEGATIVE_KEYWORDS.some(k => body.includes(k));
  });

  const unresolvedNegative = negativeComments.filter(c => !c.resolvedAt);
  const recencyDays = event.reviewAgeDays;

  // Recency-weighted negative comment count
  // Recent unresolved negatives are most painful
  const recencyMultiplier = recencyDays > 7 ? 1.5 : 1.0;
  const negativeCount = unresolvedNegative.length * recencyMultiplier;

  // Sentiment average of all comments (normalized from -100..100 to 0..100)
  const avgSentiment = event.comments.length > 0
    ? event.comments.reduce((sum, c) => sum + c.sentimentScore, 0) / event.comments.length
    : 0;

  // Invert: negative sentiment → high pain
  // sentiment ranges -100..+100; we map -100→100 (pain), +100→0 (no pain)
  const sentimentPain = Math.round((100 - avgSentiment) / 2);

  // Combine: comment-count pain + sentiment pain
  const commentPain = Math.min(50, negativeCount * 15);
  return Math.min(100, sentimentPain + commentPain);
}
```

### Signal 3: Process Violation Score (weight: 0.40)

```typescript
const SECURITY_LABELS = ['security', 'CVE', 'vulnerability', 'auth', 'crypto'];

function deriveProcessViolationScore(event: ReviewEvent): number {
  let score = 0;

  // No tests on non-trivial PR
  if (!event.hasTests && event.filesChanged.length > 2) {
    score += 35;
  }

  // Breaking change without notification
  if (event.isBreakingChange && !event.labels.some(l => l.includes('breaking'))) {
    score += 40;
  }

  // Security-sensitive changes without security review
  const hasSecurityChanges = event.filesChanged.some(f =>
    SECURITY_LABELS.some(s => f.path.toLowerCase().includes(s))
  );
  if (hasSecurityChanges && !event.hasSecurityReview) {
    score += 50; // Most severe
  }

  // Large diff with no tests
  if (event.totalLinesAdded + event.totalLinesDeleted > 500 && !event.hasTests) {
    score += 20;
  }

  return Math.min(100, score);
}
```

### Combined Score Derivation

```typescript
function deriveReviewPainScore(event: ReviewEvent): number {
  const complexity = deriveComplexityScore(event);   // 0-100
  const sentiment  = deriveSentimentScore(event);    // 0-100
  const process     = deriveProcessViolationScore(event); // 0-100

  const score = Math.round(
    0.25 * complexity +
    0.35 * sentiment  +
    0.40 * process
  );

  // Clamp
  return Math.max(0, Math.min(100, score));
}
```

**Rationale for weights:** Process violations (40%) are most impactful — they represent policy failures that affect the whole team. Sentiment (35%) captures team dynamics and morale. Diff complexity (25%) is important but least urgent.

## 3. Gap Analysis: PainSignalSchema vs Review Domain

### Required Fields — Can We Fill All?

| PainSignal Field | Review Source | Gap? |
|---|---|---|
| `source` | Map trigger type (e.g., `'process_violation'`) | No |
| `score` | deriveReviewPainScore() | No |
| `timestamp` | `new Date().toISOString()` at capture | No |
| `reason` | Build from trigger + top contributor | No |
| `sessionId` | `ReviewEvent.sessionId` | No |
| `agentId` | `'code-review-evaluator'` (fixed) | No |
| `traceId` | `ReviewEvent.traceId ?? 'unknown'` | No |
| `triggerTextPreview` | First 200 chars of top-negative comment body | No |
| `domain` | `'code-review'` (fixed) | No |
| `severity` | `deriveSeverity(score)` | No |
| `context` | Flexible `Record<string, unknown>` | No |

**Conclusion: No schema changes required.** PainSignalSchema is truly universal. The `context` field (type: `Record<string, unknown>`) absorbs all review-specific data (file list, sentiment values, violation types).

### Context Fields to Include

```typescript
context: {
  // Trigger identification
  primaryTrigger: ReviewPainTriggerType;
  diffComplexityScore: number;
  sentimentScore: number;
  processViolationScore: number;

  // Participant data
  authorId: string;
  reviewerIds: string[];
  affectedFileCount: number;
  affectedFiles: string[];  // Truncated to first 10 paths

  // Comment data
  totalCommentCount: number;
  unresolvedThreadCount: number;
  avgSentiment: number;

  // PR metadata
  hasTests: boolean;
  isBreakingChange: boolean;
  reviewAgeDays: number;
}
```

### Potential Gap: triggerTextPreview for Process Violations

When the pain source is a process violation (no tests), there is no "bad text" to preview. The adapter should use a placeholder:

```typescript
triggerTextPreview: topNegativeComment ?? `Process violation: ${primaryTrigger}`,
```

## 5. DefaultPrincipleInjector with domain='code-review'

From `principle-injector.ts`, `DefaultPrincipleInjector` operates on:

```typescript
export interface InjectionContext {
  domain: string;
  sessionId: string;
  budgetChars: number;
}
```

**Key behaviors:**

1. **Domain is informational only** — `DefaultPrincipleInjector` does NOT filter principles by domain. It sorts by priority, then by `createdAt`. Domain in `InjectionContext` is passed through but not used for selection.

2. **P0 forced inclusion** — All P0 principles are always included, regardless of `budgetChars`.

3. **Budget-aware fitting** — P1/P2 principles are added until `budgetChars` is exhausted.

**Implication for code-review:** When injecting with `domain='code-review'`, the injector will include ALL principles (across all domains) sorted by priority, not just code-review-domain principles. This is correct behavior — P0 principles are universal, and the LLM prompt benefits from seeing cross-domain wisdom.

**Test to validate:**

```typescript
// In injector conformance test, add domain='code-review' variant
const ctx = createTestContext({ domain: 'code-review', budgetChars: 200 });
const result = injector.getRelevantPrinciples(principles, ctx);
// Should include P0 (forced), may include P1/P2 if budget allows
// Should NOT filter by domain — domain is metadata only
```

## 7. Open Questions & Edge Cases

1. **Non-failure equivalent for reviews:** Unlike tool calls (success vs. failure), code review is always somewhat evaluative. The "non-failure" event for conformance testing could be a PR with zero comments and no process violations — a score near zero but still a valid PainSignal.

2. **Sentiment scoring strategy:** The algorithm above uses a simple keyword approach. Production implementations might call an LLM or use an external sentiment API. The adapter interface does not prescribe how sentiment is computed — only that `ReviewComment.sentimentScore` exists.

3. **Breaking change detection:** Labels are unreliable (depends on team conventions). The `isBreakingChange` flag should come from a build/CI signal or PR metadata, not inferred.

4. **Score threshold for "real" pain:** Should CodeReviewPainAdapter return null when score < some threshold? Current adapters (OpenClaw, Writing) do NOT filter by score — they pass through any valid signal. Following D-02, we preserve this behavior.
