**Test Setup (beforeEach/afterEach):**
- Create temp dir with `fs.mkdtempSync(path.join(os.tmpdir(), 'pd-adapter-'))`
- Create stateDir = path.join(tempDir, '.state') and `fs.mkdirSync(stateDir, { recursive: true })`
- Create a fresh `new PrincipleTreeLedgerAdapter({ stateDir })` for each test
- Cleanup with `safeRmDir(tempDir)` in afterEach

**Factory function — `createEntry(overrides): LedgerPrincipleEntry`:**
- Returns a complete 11-field LedgerPrincipleEntry with all fields populated (including optional artifactRef, taskRef)
- Uses a fixed UUID v4 for id field
- Uses `sourceRef: 'candidate://test-candidate-001'` as default

**Test blocks (each in a `describe` or individual `it`):**

**1. writeProbationEntry — happy path**
- Call `adapter.writeProbationEntry(createEntry())`
- Verify return value is the same entry (passthrough)
- Load ledger via `loadLedger(stateDir)` and verify `tree.principles[entry.id]` exists
- Verify the written principle has: `id` matches entry.id, `text` matches entry.text, `status: 'candidate'` (not 'probation'), `triggerPattern` matches entry.triggerPattern, `action` matches entry.action

**2. writeProbationEntry — idempotency**
- Call `adapter.writeProbationEntry(entry)` twice with the same entry
- Verify the second call returns the same entry (reference equality or deep equality)
- Load ledger and verify only ONE principle exists — `Object.keys(store.tree.principles).length === 1`
- Verify the ledger file only contains one entry for this id

**3. writeProbationEntry — different candidates produce different entries**
- Create two entries with different `sourceRef` values (candidate://alpha, candidate://beta) and different ids
- Write both
- Verify ledger contains 2 principles
- Verify existsForCandidate returns correct entry for each

**4. existsForCandidate — returns entry for known candidate**
- Write an entry first: `adapter.writeProbationEntry(createEntry())`
- Call `adapter.existsForCandidate('test-candidate-001')`
- Verify returned entry is not null and has correct id, title, sourceRef

**5. existsForCandidate — returns null for unknown candidate**
- Call `adapter.existsForCandidate('nonexistent-candidate')`
- Verify returns null

**6. Field expansion — status mapping**
- Write entry with `status: 'probation'`
- Load ledger and verify written principle has `status: 'candidate'`
- Verify the written principle does NOT have status 'probation'

**7. Field expansion — defaults applied**
- Write a minimal entry (only required fields: id, title, text, status, evaluability, sourceRef, createdAt — no triggerPattern, no action, no optional fields)
- Load ledger and verify: `version: 1`, `priority: 'P1'`, `scope: 'general'`, `valueScore: 0`, `adherenceRate: 0`, `painPreventedCount: 0`, `derivedFromPainIds` is an array, `ruleIds: []`, `conflictsWithPrincipleIds: []`, `createdAt` and `updatedAt` match entry.createdAt

**8. Field expansion — triggerPattern and action passthrough (Decision B)**
- Write entry with `triggerPattern: 'file delete'` and `action: 'verify backup'`
- Load ledger and verify written principle has `triggerPattern: 'file delete'` and `action: 'verify backup'`
- Write entry with `triggerPattern: undefined` and `action: undefined` (optional fields omitted)
- Load ledger and verify written principle has `triggerPattern: ''` and `action: ''`

**9. sourceRef/artifactRef/taskRef NOT in ledger (Decision C, Q3 resolved)**
- Write entry with all three provenance fields populated
- Load ledger and get the written principle from `tree.principles`
- Verify the principle object does NOT have `sourceRef` property
- Verify the principle object does NOT have `artifactRef` property
- Verify the principle object does NOT have `taskRef` property
- (The adapter Map still tracks these — verify existsForCandidate works correctly)

**10. derivedFromPainIds populated with candidateId (Q1 resolved)**
- Write entry with sourceRef `candidate://test-candidate-001`
- Load ledger and verify `derivedFromPainIds: ['test-candidate-001']`
- NOTE: `source` field must NOT be set — only `derivedFromPainIds` (Principle interface has no `source` field per RESEARCH.md Pitfall 3)

**11. Error handling — LEDGER_WRITE_FAILED on write failure**
- Use a stateDir path that would cause a write failure (e.g., a file where a directory is expected, or mock addPrincipleToLedger behavior)
- Suggestion: create a FILE at `path.join(stateDir, 'principle_training_state.json')` that is NOT a directory, which will cause the directory creation inside `ensureParentDir` to fail when it tries to `mkdirSync` on the file's path — no, that won't work because ensureParentDir calls path.dirname on the filePath

- Verify the error message contains the entry id

**12. Adapter instance isolation — separate instances have separate Maps**
