# Phase m7-02: PrincipleTreeLedger Adapter — Context

**Gathered:** 2026-04-26

<domain>
## Phase Boundary

**Domain:** Implement `LedgerAdapter` in `openclaw-plugin` that satisfies the interface defined in `principles-core`. The adapter bridges the 11-field `LedgerPrincipleEntry` contract (principles-core) to the full `LedgerPrinciple` shape (18+ fields, openclaw-plugin). It handles field expansion, ledger write via `addPrincipleToLedger()`, and candidate-level idempotency via `existsForCandidate()`.

**Scope:**
- `PrincipleTreeLedgerAdapter` class implementing `LedgerAdapter` interface
- Field expansion: 11-field `LedgerPrincipleEntry` → 18+ field `LedgerPrinciple`
- Write path: call `addPrincipleToLedger(stateDir, ledgerPrinciple)`
- Idempotency path: `existsForCandidate` using in-memory Map
- Adapter lives in `packages/openclaw-plugin/src/core/principle-tree-ledger-adapter.ts`
- Tests live in `packages/openclaw-plugin/tests/core/principle-tree-ledger-adapter.test.ts`

**Non-goals:**
- No `CandidateIntakeService` (that is m7-03)
- No CLI (that is m7-04)
- No promotion to active principle
- No pain signal bridge

</domain>

<decisions>
## Implementation Decisions

### A — LedgerPrinciple.id format

**Decision: Direct UUID v4 reuse.**
`LedgerPrincipleEntry.id` (UUID v4 generated by the intake service) is used directly as `LedgerPrinciple.id` in the ledger. No sequential `P_XXX` scanning. No conflict with existing `P_XXX` entries in the ledger — mixed ID formats in `tree.principles` are acceptable.

The `sourceRef = 'candidate://<candidateId>'` is the primary provenance/idempotency key (D-11 from m7-01) and is tracked separately (see C below).

### B — triggerPattern / action defaults

**Decision: Extract from DiagnosticianRecommendation JSON; null if absent.**
`CandidateRecord.sourceRecommendationJson` contains the full `DiagnosticianOutputV1` JSON. The adapter parses it to extract `triggerPattern` and `action` from the `DiagnosticianRecommendation` record.

- If `triggerPattern` is present in the recommendation → use it
- If absent → `triggerPattern` field is omitted from `LedgerPrincipleEntry` (Type.Optional, null equivalent)
- Same logic for `action`

No derivation from `description` text. M7 does not infer patterns.

### C — sourceRef storage in ledger

**Decision: sourceRef is NOT written to the ledger file.**
`LedgerPrinciple` in `principle-tree-schema.ts` has no `sourceRef` field. The `sourceRef = 'candidate://<candidateId>'` is maintained at the adapter memory layer only (see D).

This means the ledger file itself does not contain candidate provenance. The provenance chain is: `CandidateIntakeService` → adapter Map → `LedgerPrinciple.id` (UUID). Downstream consumers trace via `LedgerPrinciple.id` + `artifactRef` (which IS written to the ledger as an annotation or metadata).

### D — existsForCandidate implementation

**Decision: Adapter instance-level `Map<candidateId, LedgerPrincipleEntry>` for idempotency tracking.**
The adapter maintains an in-memory `Map` that tracks `candidateId → LedgerPrincipleEntry` for all written entries.

**Contract:**
- `writeProbationEntry`:
  1. Consult in-memory Map — if `candidateId` already mapped, return existing entry (no-op, idempotent)
  2. If absent: expand `LedgerPrincipleEntry` → `LedgerPrinciple` (apply defaults, see field expansion block below)
  3. Call `addPrincipleToLedger(stateDir, ledgerPrinciple)`
  4. Record mapping in in-memory Map
  5. Return the input `LedgerPrincipleEntry`
- `existsForCandidate(candidateId)`: consult in-memory Map → return entry or null

**Singleton requirement:** The adapter instance must be reused across calls. Caller (`CandidateIntakeService` in m7-03, or CLI handler in m7-04) should instantiate the adapter once and pass the same instance. This is consistent with the DI pattern from D-07 (m7-01 CONTEXT).

### Field Expansion — 11 fields → LedgerPrinciple (18+ fields)

When `writeProbationEntry` expands a `LedgerPrincipleEntry` to a full `LedgerPrinciple`, it applies these defaults:

```
LedgerPrincipleEntry (11 fields)  →  LedgerPrinciple (18+ fields)
────────────────────────────────       ────────────────────────────────
id: UUID v4                        →  id: UUID v4 (same)
title: string                      →  text: "When {triggerPattern}, then {action}." OR title if no trigger/action
text: string                       →  text: same (principle statement)
triggerPattern: string | null      →  trigger: triggerPattern (or "")
action: string | null              →  action: action (or "")
status: 'probation'                →  status: 'candidate' (ledger uses 'candidate' not 'probation')
evaluability: 'weak_heuristic'     →  evaluability: 'weak_heuristic'
sourceRef: candidate://<id>        →  (NOT written to ledger — adapter memory only)
artifactRef: artifact://<id>        →  (stored as annotation or provenance metadata)
taskRef: task://<id> | absent      →  (stored as provenance metadata, optional)
createdAt: ISO timestamp           →  createdAt: same, also used as updatedAt initially

--- Fields set to defaults (m7-02 does not populate) ---
version: 1
priority: 'P1' (default)
scope: 'general' (default)
domain: undefined
valueScore: 0
adherenceRate: 0
painPreventedCount: 0
derivedFromPainIds: []
ruleIds: []
conflictsWithPrincipleIds: []
supersedesPrincipleId: undefined
updatedAt: createdAt
deprecatedAt: undefined
deprecatedReason: undefined
detectorMetadata: undefined
compilationRetryCount: undefined
suggestedRules: undefined
source.painId: candidateId (as proxy, for compatibility)
source.painType: 'diagnostician'
source.timestamp: createdAt
```

**Important: `status: 'probation'` in LedgerPrincipleEntry maps to `status: 'candidate'` in LedgerPrinciple.** The ledger uses `'candidate'` as the probationary status, not `'probation'`.

</decisions>
