# Phase m7-02: PrincipleTreeLedger Adapter - Research

**Researched:** 2026-04-26
**Domain:** LedgerAdapter interface implementation, field expansion, file-locked JSON mutation
**Confidence:** HIGH

## Summary

This phase implements the `LedgerAdapter` interface (defined in `principles-core`, m7-01 SHIPPED) as a concrete class in `openclaw-plugin`. The adapter bridges the 11-field `LedgerPrincipleEntry` contract to the 18+ field `LedgerPrinciple` shape and writes to the existing file-based ledger via `addPrincipleToLedger()`. The adapter also maintains an in-memory `Map<candidateId, LedgerPrincipleEntry>` for idempotency tracking per the 4 locked user decisions (A-D) in CONTEXT.md.

The implementation is straightforward -- the ledger write function (`addPrincipleToLedger`) already exists, is production-hardened (atomic writes, PID-based file locks, stale lock detection), and is idempotent by key. The adapter's primary responsibility is field mapping: expanding 11 intake-level fields into the full `LedgerPrinciple` shape with sensible defaults. The `existsForCandidate` idempotency path uses an in-memory Map (not ledger scanning) per Decision D.

**Primary recommendation:** Create a single `PrincipleTreeLedgerAdapter` class in `principle-tree-ledger-adapter.ts` with constructor receiving `stateDir`. Use the `createPrincipleFromDiagnosis()` pattern in `evolution-reducer.ts` as the reference for field population (it already does the same kind of expansion from minimal fields to full `LedgerPrinciple`).

## User Constraints (from CONTEXT.md)

### Locked Decisions

#### A -- LedgerPrinciple.id format
Direct UUID v4 reuse. `LedgerPrincipleEntry.id` (UUID v4 generated by the intake service) is used directly as `LedgerPrinciple.id` in the ledger. No sequential `P_XXX` scanning. Mixed ID formats in `tree.principles` are acceptable. `sourceRef = 'candidate://<candidateId>'` is the primary provenance/idempotency key.

#### B -- triggerPattern / action defaults
Extract from `DiagnosticianRecommendation` JSON; null if absent. `CandidateRecord.sourceRecommendationJson` contains the full `DiagnosticianOutputV1` JSON. If `triggerPattern` is present in the recommendation -> use it. If absent -> the `triggerPattern` field is omitted from `LedgerPrincipleEntry` (Type.Optional, null equivalent). Same logic for `action`. No derivation from `description` text.

#### C -- sourceRef storage in ledger
sourceRef is NOT written to the ledger file. `LedgerPrinciple` in `principle-tree-schema.ts` has no `sourceRef` field. The `sourceRef = 'candidate://<candidateId>'` is maintained at the adapter memory layer only (see D). Downstream consumers trace via `LedgerPrinciple.id` + `artifactRef`.

#### D -- existsForCandidate implementation
Adapter instance-level `Map<candidateId, LedgerPrincipleEntry>` for idempotency tracking. Contract:
- `writeProbationEntry`: Consult Map -> if found, return existing (no-op, idempotent). If absent: expand -> write -> record -> return.
- `existsForCandidate(candidateId)`: Consult Map -> return entry or null.
- **Singleton requirement:** Adapter instance must be reused across calls.

### Pattern 1: Constructor-based DI with Options Object

**What:** The adapter receives `stateDir` via constructor; no global state, no service locator.

**When to use:** All adapter/service classes in openclaw-plugin follow this pattern.

**Example:**
```typescript
// Source: FileStorageAdapter pattern (file-storage-adapter.test.ts L41)
// and EvolutionReducerImpl pattern (evolution-reducer.ts L109)
export class PrincipleTreeLedgerAdapter implements LedgerAdapter {
  private readonly stateDir: string;
  private readonly entryMap = new Map<string, LedgerPrincipleEntry>();

  constructor(opts: { stateDir: string }) {
    this.stateDir = opts.stateDir;
  }

  writeProbationEntry(entry: LedgerPrincipleEntry): LedgerPrincipleEntry {
    // Extract candidateId from sourceRef
    const candidateId = extractCandidateId(entry.sourceRef);

    // Idempotency check (Decision D)
    const existing = this.entryMap.get(candidateId);
    if (existing) return existing;

    // Expand 11 → 18+ fields
    const ledgerPrinciple = this.expandToLedgerPrinciple(entry, candidateId);

    // Write to ledger
    try {
      addPrincipleToLedger(this.stateDir, ledgerPrinciple);
    } catch (err) {
      throw new CandidateIntakeError(
        INTAKE_ERROR_CODES.LEDGER_WRITE_FAILED,
        `Failed to write principle ${entry.id} to ledger: ${String(err)}`,
        { cause: err },
      );
    }

    // Record for idempotency
    this.entryMap.set(candidateId, entry);
    return entry;
  }

  existsForCandidate(candidateId: string): LedgerPrincipleEntry | null {
    return this.entryMap.get(candidateId) ?? null;
  }

  private expandToLedgerPrinciple(entry: LedgerPrincipleEntry, candidateId: string): LedgerPrinciple {
    const now = entry.createdAt;
    return {
      id: entry.id,
      version: 1,
      text: entry.text,
      trigger: entry.triggerPattern ?? '',
      action: entry.action ?? '',
      status: 'candidate',  // probabation → candidate
      evaluability: entry.evaluability,
      coreAxiomId: undefined,
      priority: 'P1',
      scope: 'general',
      domain: undefined,
      valueScore: 0,
      adherenceRate: 0,
      painPreventedCount: 0,
      lastPainPreventedAt: undefined,
      derivedFromPainIds: [],
      ruleIds: [],
      conflictsWithPrincipleIds: [],
      supersedesPrincipleId: undefined,
      createdAt: now,
      updatedAt: now,
      deprecatedAt: undefined,
      deprecatedReason: undefined,
      detectorMetadata: undefined,
      compilationRetryCount: undefined,
      suggestedRules: undefined,
    };
  }
}
```

## Common Pitfalls

### Pitfall 1: TypeScript ESM Import Paths

**What goes wrong:** Tests import from `../../src/core/principle-tree-ledger.js` but the source file is `.ts`. This works because vitest + tsconfig handle it, but using the wrong extension (`.ts` instead of `.js`) causes module resolution failures.

**Why it happens:** The project uses ESM (`"type": "module"`) where imports must use `.js` extensions referencing compiled output, not `.ts` source.

**How to avoid:** Always use `.js` extension in import paths. Follow the exact pattern from `principle-tree-ledger.test.ts`:
```typescript
import { addPrincipleToLedger, type LedgerPrinciple } from '../../src/core/principle-tree-ledger.js';
```

**Warning signs:** `ERR_MODULE_NOT_FOUND` or `Cannot find module` errors.

### Pitfall 2: stateDir Must Exist Before Write

**What goes wrong:** `addPrincipleToLedger()` calls `ensureParentDir()` which creates directories if missing, BUT test setup must ensure the stateDir exists or the ledger file path resolution may fail silently.

**Why it happens:** The `getLedgerFilePath()` simply joins paths; `ensureParentDir()` is called inside `writeLedgerUnlocked()` but not in `readLedgerFromFile()`.

**How to avoid:** Always `fs.mkdirSync(stateDir, { recursive: true })` in test `beforeEach` -- exactly as `principle-tree-ledger.test.ts` does.

**Warning signs:** Tests that pass in isolation but fail when the temp directory structure isn't pre-created.

### Pitfall 3: Principle Interface Source Field

**What goes wrong:** Attempting to set `source.painId` and `source.painType` on `LedgerPrinciple` per the CONTEXT.md field expansion table.

**Why it happens:** The `Principle` interface in `principle-tree-schema.ts` (which `LedgerPrinciple` extends) does NOT have a `source` field. The `source` field exists only on the `evolution-types.ts` `Principle` interface, which is an entirely different type used by the evolution reducer, not the tree ledger.

**How to avoid:** Use `derivedFromPainIds: []` (which exists on the tree schema `Principle`) instead of `source.painId`. The `evolution-reducer.ts` line 430 demonstrates this: `derivedFromPainIds: [params.painId]`. In M7, this should be set to `[]` since the candidate ID is not a painId -- the provenance chain is `candidateId → LedgerPrinciple.id (UUID)` tracked in the adapter Map.

**Warning signs:** TypeScript compilation error: `Object literal may only specify known properties, and 'source' does not exist in type 'LedgerPrinciple'`.

### Pitfall 4: Idempotency Map Lifetime

**What goes wrong:** Creating a new adapter instance for each intake call, breaking idempotency.

**Why it happens:** The Map is per-instance. If the caller creates a new adapter each time, `existsForCandidate()` will always return null.

**How to avoid:** Document the singleton requirement clearly. The caller (m7-03 `CandidateIntakeService` or m7-04 CLI handler) must create the adapter once and reuse it. The adapter's constructor JSDoc should state: "This adapter maintains an in-memory idempotency map. Create once and reuse across calls within the same process lifetime."

**Warning signs:** Duplicate ledger entries for the same candidateId after multiple intake calls.

## Open Questions

1. **derivedFromPainIds for candidate entries**
   - What we know: `evolution-reducer.ts` sets `derivedFromPainIds: [params.painId]` for pain-derived principles. M7 candidate intake does not have a direct painId.
   - What's unclear: Should `derivedFromPainIds` be `[]` (empty, no pain provenance) or `[candidateId]` (using candidate as proxy)? The CONTEXT.md field expansion says `source.painId: candidateId (as proxy, for compatibility)` which suggests using the candidateId. However, the field `source.painId` doesn't exist on `LedgerPrinciple` -- only `derivedFromPainIds` does. The user may intend `derivedFromPainIds: [candidateId]` as the proxy.
   - Recommendation: Set `derivedFromPainIds: [candidateId]` to match the stated intent of "candidateId as proxy, for compatibility". This is consistent with the CONTEXT.md field expansion table even though the field name differs between the table (`source.painId`) and the actual type (`derivedFromPainIds`).

2. **Should the adapter parse `sourceRecommendationJson` to extract triggerPattern/action?**
   - What we know: Decision B says "Extract from DiagnosticianRecommendation JSON; null if absent." But the adapter receives `LedgerPrincipleEntry` which already has `triggerPattern` and `action` fields populated by the intake service (m7-03).
   - What's unclear: Does the adapter need to parse `sourceRecommendationJson` itself, or does it rely on `LedgerPrincipleEntry` fields already populated by m7-03? The CONTEXT.md wording "The adapter parses it" suggests the adapter does the parsing directly, but the entry schema already has optional `triggerPattern`/`action` fields.
   - Recommendation: The adapter should NOT parse `sourceRecommendationJson`. The `LedgerPrincipleEntry` fields are the contract. If `triggerPattern` is present on the entry, use it. If absent, use `''` (empty string). This keeps the adapter simple and avoids tight coupling to `DiagnosticianRecommendation` structure. The parsing responsibility belongs to m7-03 (CandidateIntakeService). Flag this for user confirmation in discuss phase.

3. **artifactRef / taskRef storage in ledger**
   - What we know: The CONTEXT.md says `artifactRef` and `taskRef` should be "stored as provenance metadata." The `LedgerPrinciple` type has no dedicated field for these.
   - What's unclear: Where exactly do these go? Options: (a) `suggestedRules` array as string annotations, (b) custom metadata field that the type doesn't have, (c) omit entirely since `sourceRef` is the idempotency key and the adapter Map tracks it.
   - Recommendation: Omit `artifactRef` and `taskRef` from the ledger entry since `LedgerPrinciple` has no corresponding fields. The provenance chain is maintained via: `candidateId -> adapter Map -> LedgerPrinciple.id (UUID)`. Downstream consumers can trace via the adapter Map or the `sourceRef` field in the Map. This is consistent with Decision C (sourceRef not written to ledger).
